# 项目与章节需求

## 目标

支持中文小说作者创建作品、管理章节、稳定编辑正文，并在章节完成前执行必要的一致性终检。

## 小说项目

### 创建小说

用户可以在工作台首页创建小说，输入小说名和简介。

业务规则：

- 用户必须登录。
- 小说名不能为空；为空时不创建。
- 新建小说自动创建：
  - 第一章；
  - 空文本总纲；
  - 默认剧情进度，当前阶段为“开篇”。
- 小说归属当前用户。
- 创建后首页需要刷新，并返回小说 ID 和第一章 ID。

### 工作台首页小说列表

工作台首页 `/dashboard` 展示当前登录用户的小说列表。根路由 `/` 是官网首页，负责产品说明、登录入口和注册入口。

展示信息：

- 小说名；
- 简介；
- 章节数量；
- 已应用文风；
- 更新时间。

排序规则：

- 按小说 updatedAt 倒序展示。

### 进入工作台

工作台按小说 ID 进入，并读取当前用户有权访问的小说。

默认章节选择规则：

1. URL 指定 chapterId 时优先使用该章节。
2. 否则选择最后一个仍处于 drafting 的章节。
3. 否则选择最后一章。

## 章节管理

### 新增章节

用户可以在章节列表新增章节。

业务规则：

- 必须校验小说归属。
- 新章节 order 为当前最大 order 加 1。
- 默认标题为“第 N 章”。
- 新增后刷新工作台。

### 章节列表

章节列表按 order 升序展示。

展示信息：

- 章节标题；
- 章节顺序；
- 章节状态；
- 更新时间。

章节状态标签：

- drafting：草稿中；
- review：待审核；
- completed：完成。

## 章节正文编辑器

### 编辑标题和正文

用户可以编辑当前章节标题和正文。

业务规则：

- 标题为空时保存为“未命名章节”。
- 正文保存到 Chapter.content。
- 编辑器使用 textarea。
- 自动保存延迟为 1.2 秒。
- 标题和正文只有在章节为 drafting 时可编辑；review 和 completed 必须先退回草稿。
- 同一章节同一时刻只允许一个保存请求；保存期间的新输入在当前请求结束后立即按最新快照继续保存。
- 保存和状态切换必须携带 `expectedUpdatedAt`，Core 锁定章节后拒绝过期版本；相同标题和正文的重试按幂等成功处理。
- 保存成功后返回单调递增的更新时间，客户端只在请求成功后推进本地版本。
- 切章、新建章节和送审前必须先 flush 当前草稿；保存失败或版本冲突时保留输入并阻止动作。
- 未落盘草稿按用户、小说和章节隔离保存到浏览器本地存储，刷新或异常离开后允许恢复；浏览器拒绝访问、读写或清理本地存储时不能阻断服务端自动保存。版本冲突只在确认本地草稿已清除后重新加载，清理失败时保留正文并明确提示。

### 字数统计

系统按去除 Unicode 空白字符和 `U+FEFF` 后的 Unicode 码点长度统计字数。浏览器可见字数统一调用 `countTextLength()`，Core 的 Python 计算和 PostgreSQL 聚合使用同一排除集合。

展示范围：

- 当前章节字数；
- 当前小说总字数。

### 章节进展

用户可以为章节维护独立的章节进展记录。

业务规则：

- 内容保存到 ChapterProgress。
- 如果记录不存在，则自动创建。
- 章节进展用于记录本章关键事件、伏笔、进度，不等同于正式正文。

## 章节送审与完成

章节状态流转：

~~~mermaid
stateDiagram-v2
    [*] --> drafting
    drafting --> review: 用户点击送审
    review --> drafting: 用户撤回到草稿
    review --> completed: 一致性终检完成或跳过后标记完成
    completed --> review: 用户重新送审
    completed --> drafting: 用户改回草稿
~~~

### 送审

用户点击“送审”后，章节进入 review。

业务规则：

- 送审或完成时，系统确保默认章节质量检查项存在。
- 当前默认检查项只有一致性终检。

### 一致性终检

送审后，用户可以运行、跳过或重置一致性终检。

正文实际变化时，旧检查结果立即失效：检查项重置为 pending，活动质量运行标记 cancelled；只修改标题不使正文检查失效。质量运行绑定发起时的完整正文快照和 SHA-256，旧正文的延迟回调不能完成当前检查项。只有 review 章节可以创建质量运行；活动运行期间不能重置或跳过，completed 章节不能再修改检查状态。前端在运行受理后轮询到终态，pending 或 running 时禁止重复运行或跳过；轮询失败或超时后允许继续查询同一运行。

检查内容：

- 正文与设定是否冲突；
- 角色是否 OOC；
- 世界规则是否自洽；
- 伏笔是否误用；
- 剧情逻辑是否矛盾。

状态：

- pending：待处理；
- running：运行中；
- completed：完成；
- skipped：跳过；
- failed：失败。

### 标记完成

章节标记 completed 前，必须满足：

- 一致性终检存在；
- 且状态为 completed 或 skipped。

完成后：

- Chapter.completedAt 写入当前时间；
- 工作台刷新。

## 章节质量检查流程

~~~mermaid
flowchart TD
    A["用户编辑正文"] --> B["自动保存章节"]
    B --> C["用户点击送审"]
    C --> D["创建或更新一致性终检项"]
    D --> E{"用户处理终检"}
    E -->|"运行"| F["调用质量检查 API"]
    E -->|"跳过"| G["状态改为 skipped"]
    F --> H["校验 Agent 生成报告"]
    H --> I["报告保存到 ChapterQualityCheck"]
    I --> J["状态 completed"]
    G --> K["允许标记完成"]
    J --> K
    K --> L["用户标记章节完成"]
~~~

## 权限与安全

- 章节读写必须通过章节归属找到 novelId，再校验当前用户是否拥有该小说。
- 质量检查项运行必须校验检查项所属章节和小说归属。
- 如果传入 taskId，还必须校验 task.chapterId 与 check.chapterId 一致。

## 验收标准

- 登录作者可以创建小说，并自动拥有第一章。
- 根路由 `/` 显示官网首页，工作台首页 `/dashboard` 显示作者小说列表。
- 作者可以新增章节、切换章节、编辑标题和正文。
- 正文编辑后自动保存，刷新后内容保留。
- 并发窗口中的过期正文写入返回版本冲突，不能覆盖较新的服务端正文。
- 自动保存失败、切章前 flush 失败或版本冲突时，正文仍保留且界面不能伪装为已保存。
- 字数统计与 countTextLength 规则一致。
- 送审后能看到一致性终检入口。
- 未完成或跳过一致性终检时不能标记章节完成。
- 正文变化后，旧的一致性报告、跳过状态和旧正文回调都不能继续满足完成门禁。
- 越权访问他人小说、章节、质量检查项会被拒绝。
