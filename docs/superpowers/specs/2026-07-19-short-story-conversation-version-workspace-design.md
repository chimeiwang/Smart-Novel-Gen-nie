# 中短篇会话与版本工作台设计

## 状态

- 日期：2026-07-19
- 状态：已确认
- 范围：中短篇工作区布局、写作会话、大纲版本、正文版本、版本引用与批准后继续修改
- 数据库约束：不修改 PostgreSQL schema，不新增数据库枚举

## 背景

当前中短篇工作区虽然能够生成完整大纲、保存 revision、生成整稿并完成审核，但界面把作品信息、改纲输入、审核操作和版本历史全部堆在右栏。它没有复用长篇已经具备的 WritingSession 会话模型，不能开始新对话或选择旧对话继续，也把“大纲批准”错误地表现为无法继续修改的终点。

中短篇需要同时处理三类彼此独立的对象：

1. 会话：用户和 Agent 围绕作品讨论、比较或提出修改；
2. 版本：大纲与正文每次实际变化形成的不可变历史快照；
3. 当前采用状态：用户当前认可的大纲版本和正式正文版本。

会话不是版本，批准也不是删除历史。用户必须可以在批准后新开会话或接续旧会话，引用任意历史大纲或正文版本继续讨论和微调。

## 目标

- 中短篇工作区采用“左侧进度与版本、中间作品与内容、右侧纯对话”的稳定三栏结构。
- 复用长篇 WritingSession 的会话语义，支持开始新对话、查看历史对话和继续原对话。
- 普通讨论不自动产生草案；明确要求修改时才生成新的大纲或正文版本。
- 大纲与正文分别使用项目级、单调递增且可稳定引用的版本号。
- 任何历史版本都可以在中栏查看，也可以附加到聊天中作为精确上下文。
- 用户在文字中明确说“大纲版本 2”“正文 v3”等引用时，Agent 能读取该版本的完整权威内容。
- 批准只改变“当前采用版本”，不关闭作品的后续修改能力，不覆盖或删除历史版本。
- 继续使用中短篇专用完整大纲、整稿生成、串行双审核和单 Chapter 正式正文规则。

## 非目标

- 不把大纲分节转换为 Chapter、Beat Plan 或独立任务。
- 不把正文拆成逐节生成流程。
- 不把长篇的章节列表、Beat Plan 或章节审核栏搬进中短篇工作区。
- 不把几十轮历史会话全部注入每次模型上下文。
- 不允许模糊版本引用静默回退到最新版本。
- 不修改 PostgreSQL 表、字段、枚举或迁移。

## 1. 页面结构

### 1.1 左栏：进度与版本

左栏分为三个区域：

1. 写作进度：大纲讨论、正文创作、全稿审核、正式正文；
2. 大纲版本：按项目级版本号倒序显示；
3. 正文版本：按项目级版本号倒序显示，并展示来源大纲版本。

每个版本项至少展示：

- 人类可读版本号，例如“大纲 v3”“正文 v2”；
- 状态：当前采用、待确认、历史版本或生成失败；
- 修改摘要；
- 创建时间；
- 来源会话；
- 正文版本额外展示来源大纲版本。

点击版本会在中栏打开完整内容。版本项提供“在对话中引用”，把稳定版本引用加入右侧当前会话输入区。左栏不展示长篇章节导航。

### 1.2 中栏：作品信息与完整内容

中栏顶部固定为作品信息与当前内容工具条：

- 可修改标题；
- 可选篇幅参考；
- 当前查看对象及版本号；
- 大纲、正文、正式正文切换；
- 直接编辑、批准、采用为当前版本、生成整稿等与当前内容直接相关的操作。

中栏主体只承担完整内容阅读、编辑、历史版本预览和正文审核结论。大纲仍一次展示完整人类可读内容；正文仍展示完整稿件。审核结论归属于具体正文版本，在该版本详情中展示，不进入右侧聊天之外的检查器。

### 1.3 右栏：纯对话

右栏只包含：

- 当前会话标题；
- 历史对话入口；
- 开始新对话；
- 当前会话消息流；
- 本轮引用的大纲或正文版本标签；
- 输入框、发送状态和错误反馈。

作品信息、版本列表、批准按钮、审核列表和恢复版本操作不得出现在右栏。

## 2. 会话模型

### 2.1 开始新对话

“开始新对话”创建新的 WritingSession。新会话不复制旧会话消息，默认固定以下基准：

- 当前采用的大纲版本；
- 当前正式正文版本；
- 原始灵感和创作锚点；
- 作品当前标题、文风和可选篇幅参考。

基准版本引用写入现有 WritingMessage metadata 或会话的首条 system 消息 metadata，不新增数据库字段。

### 2.2 继续历史对话

用户从历史对话列表选择会话后，恢复该 WritingSession 的完整可见消息和会话绑定任务。继续发送仍写入同一 session，不创建复制会话，也不把其他会话历史混入上下文。

会话列表复用长篇现有交互：可查看标题、阶段、最后更新时间和最后消息摘要；当前阶段存在活动任务时继续恢复其任务状态。

### 2.3 普通讨论与修改请求

中短篇自由聊天允许三种结果：

| 意图 | 行为 |
| --- | --- |
| 普通讨论、比较、询问 | 只返回 Agent 消息，不创建版本 |
| 修改大纲 | 生成新的完整大纲版本，进入待确认 |
| 修改或生成正文 | 生成新的完整正文版本，执行中短篇全稿审核后进入待确认 |

自由聊天入口使用与长篇相同的模型分类思想，但路由集合限制为中短篇允许的讨论、大纲和正文操作。不得用前端关键词匹配猜测操作。最终 Operation、引用版本和用户原文必须写入持久命令和稳定快照。

## 3. 项目级版本模型

### 3.1 大纲版本

每次大纲可见内容实际变化时创建下一个项目级大纲版本：v1、v2、v3……。状态切换、相同内容重放和幂等重试不增加版本。

大纲版本包含：

- 稳定版本引用；
- 项目级版本号；
- 完整 ShortStoryOutlineDraft；
- 修改摘要与用户原始要求；
- 来源会话和任务；
- 创建时间；
- 是否为当前采用版本。

已有 ReviewArtifactRevision 继续作为不可变版本载体。跨会话重新修改时，Core 复用项目级中短篇大纲 Artifact，并把当前任务绑定更新到新运行；revision 因而在会话之间继续单调递增。历史 ReviewArtifactRevision 不变。

### 3.2 正文版本

每次完整正文实际变化时创建下一个项目级正文版本。正文版本包含：

- 稳定版本引用；
- 项目级版本号；
- 完整 ShortStoryChapterDraft；
- 来源大纲 Artifact ID、revision、hash 和人类可读大纲版本号；
- 来源会话和任务；
- 审核结论；
- 创建时间；
- 是否为当前正式正文。

Core 复用项目级中短篇正文 Artifact，使正文 revision 在不同会话和不同返工轮次之间保持单调递增。正式 Chapter 仍只保存当前采用正文内容；历史正文继续保留在 ReviewArtifactRevision 中。

### 3.3 批准语义

- 批准大纲：把精确大纲版本标记为当前采用版本，并允许基于它生成正文。
- 批准正文：把精确正文版本写入唯一正文 Chapter，并标记为当前正式正文。
- 批准不删除历史、不锁死会话，也不禁止继续生成后续版本。
- 用户批准后继续修改时，旧版本保持可查看、可引用；新修改产生更高版本号。
- 新大纲版本不会立即覆盖正式正文。正文版本始终记录来源大纲版本，直到用户批准新的正文版本，正式 Chapter 才变化。

当前采用版本信息继续使用现有 Artifact 状态、正式 Chapter 内容和 JSON 元数据表达，不新增数据库字段。Core 聚合响应必须明确区分“最新版本”“当前采用版本”和“当前正式正文版本”。

## 4. 精确版本引用

### 4.1 显式附加

用户点击左侧“在对话中引用”后，输入框上方显示不可歧义的引用标签，例如：

```text
引用：大纲 v2
引用：正文 v1
```

发送请求携带强类型版本引用：kind、artifactId、revision 和内容 hash。Core 校验版本存在、归属当前作品且 hash 匹配，再写入 WritingMessage、WritingRunCommand 和稳定快照。

### 4.2 自然语言引用

每次中短篇会话上下文提供精简版本索引，包含类型、版本号、状态、摘要和稳定引用，不包含所有完整内容。新增只读 Agent 工具按精确类型和版本号读取完整历史版本。

当用户说“大纲版本 2”“大纲 v2”“正文版本 3”时：

1. Agent 从版本索引解析唯一目标；
2. 调用只读版本工具读取完整权威载荷；
3. 在回复或新版本元数据中记录实际引用；
4. 版本不存在、类型不清或出现冲突时停止修改并要求澄清，不能静默使用最新版。

实施时进一步收紧为 Core 前置解析：对“大纲 v2”“大纲第 2 版”“正文
v3”等无歧义表达，Core 在创建任务前直接解析项目级版本索引、校验
Artifact/revision/hash，并把完整强类型载荷放入本轮上下文。这样缺失或冲突能在
模型调用前失败，也不要求 Agent 先猜测再调用工具；显式附加与文字引用最终使用
同一种 `ShortStoryVersionReference`。

### 4.3 上下文优先级

每轮上下文按以下优先级裁决：

1. 用户本轮修改或讨论要求原文；
2. 本轮前端显式附加的版本；
3. 本轮文字中明确引用并由工具解析的版本；
4. 用户本轮已经保存的直接编辑与明确改变的锚点；
5. 当前会话固定的基准版本；
6. 当前采用的大纲和正式正文版本；
7. 其他已确认创作锚点与原始灵感；
8. 当前会话最近消息，仅用于理解指代。

高优先级版本内容是本轮修改基线。最近对话不能覆盖精确版本引用。

## 5. 运行与兼容

- 首次创建中短篇仍自动创建 WritingSession 并生成大纲 v1。
- 旧中短篇项目读取时，把现有大纲和正文 ReviewArtifactRevision 直接映射为项目级版本，不修改历史内容。
- 新会话启动大纲或正文操作时，Core 在模型调用前锁定对应项目级 Artifact、参考版本和当前任务，避免两个会话并发覆盖。
- 项目同一时刻继续只允许一个模型任务执行；其他会话可以读取历史，但发送修改请求时得到明确忙碌状态。
- 中短篇 Artifact 在新会话中再次写入时允许由 `applied` 进入新的生成/待确认周期，但必须来自新的持久命令和任务；通用长篇 ReviewArtifact 状态机不放宽。
- 新正文版本必须使用用户指定或当前采用的大纲版本作为来源，并继续执行完整性检查、编辑审核、校验审核和最多一次自动返工。
- Agent Service 仍不连接 PostgreSQL，只通过 Core 内部工具读取版本和提交草案。

## 6. 公共契约

新增或扩展以下契约，先修改 FastAPI/Pydantic 和共享契约，再生成前端客户端：

- `ShortStoryVersionReference`：kind、artifactId、revision、hash；
- `ShortStoryVersionListItem`：版本号、状态、摘要、来源会话、来源版本和时间；
- `ShortStoryVersionDetail`：版本列表字段加完整强类型 payload 与审核结论；
- 中短篇工作区聚合：最新大纲版本、当前采用大纲版本、最新正文版本、当前正式正文版本；
- 写作启动和消息请求：显式附加的版本引用；
- 项目级大纲/正文版本列表与详情接口；
- 中短篇历史版本只读内部工具接口。

版本 DTO 必须由生成的 OpenAPI 客户端提供，前端不得手写重复 DTO。

## 7. 错误处理

- 引用不存在、归属错误、hash 不一致：模型调用前返回明确冲突。
- 指定版本在发送前被删除或替换：保留输入和引用标签，提示重新选择。
- 会话存在活动任务：其他会话禁止启动修改，但允许只读查看。
- 新版本生成失败：旧版本、当前采用版本和正式正文保持不变。
- 版本提交成功但前端刷新失败：从持久 WritingSession、WritingRunCommand 和 ReviewArtifactRevision 恢复。
- 大纲版本变化不会把旧正文伪装成基于新大纲；左栏必须继续显示真实来源关系。

## 8. 验收标准

- 左栏同时显示写作进度、大纲版本和正文版本；右栏只显示会话相关内容。
- 标题和篇幅参考位于中栏顶部。
- 可以开始两个独立新对话，并分别恢复、继续发送消息。
- 普通讨论不会创建大纲或正文版本。
- 在已批准大纲和已应用正文之后仍可以新开会话并生成后续版本。
- 大纲与正文版本号分别单调递增，刷新和跨会话后不重置。
- 点击“在对话中引用大纲 v2”后，Agent 使用 v2 完整内容而不是当前最新版。
- 直接输入“比较正文 v1 和 v3”时，Agent 读取两个精确版本；不存在的版本不会静默降级。
- 新正文版本显示其真实来源大纲版本，批准前不覆盖正式 Chapter。
- 旧中短篇项目现有版本、对话和正式正文可继续读取。
- PostgreSQL schema 指纹保持不变。
