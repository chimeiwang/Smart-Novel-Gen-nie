# 章节安全保存与质量版本门禁规格

## 状态

- 日期：2026-07-15
- 状态：已实现
- 范围：章节自动保存、并发写保护、正文版本与一致性终检

## 背景

章节编辑器用 1.2 秒定时器直接发送 PATCH。切换章节会卸载组件并取消尚未触发的定时器；多个已发请求可以逆序完成；网络失败没有重试或明确错误。服务端更新又没有版本条件，因此正文可能静默丢失或回退。

一致性终检只绑定检查项和 WorkflowRun，没有绑定发起时的正文。正文变化不会废弃旧结果，运行中的旧正文报告也可能在当前正文已经变化后将公共检查项标记为完成。

## 目标

- 同一章节同一时刻最多一个正文保存请求；保存期间的新输入在前一请求结束后立即保存最新快照。
- 切换章节、送审和离开编辑状态前必须尝试 `flush()`；失败时不得伪装为已保存。
- 服务端使用现有 `Chapter.updatedAt` 拒绝过期写入，不修改 PostgreSQL schema。
- 正文变化立即废弃旧质量结果和活动运行。
- 质量运行保存正文哈希，只有同一正文版本的最新运行能够满足完成门禁。
- 质量检查 UI 能从 pending/running 自动更新到终态，并禁止重复运行或跳过。

## 非目标

- 不引入富文本编辑器、Tailwind 或全局状态库。
- 不新增章节版本历史表；本轮只保证当前正文不被无声覆盖。
- 不修改 PostgreSQL schema，不处理公网部署配置。

## 设计

### 1. 乐观并发契约

`UpdateChapterRequest` 和 `ChapterStatusRequest` 增加必填 `expectedUpdatedAt`。Web 保存当前服务端返回的 `updatedAt`，每次正文或状态 PATCH 携带该值；Core 锁定章节后比较版本，不一致时返回 409 `CHAPTER_VERSION_CONFLICT`，响应不能覆盖数据库正文。

保存和状态切换成功都返回新的 `updatedAt`，客户端仅在对应请求成功后推进本地服务端版本。所有章节写路径显式使用 `max(utc_now(), oldUpdatedAt + 1ms)`，避免数据库毫秒精度下两个写入得到同一版本。如果请求版本已旧，但数据库标题和正文已经与本次请求完全一致，则按“提交成功但响应丢失”的幂等重试返回当前 `updatedAt`。

### 2. 自动保存协调器

从 React 组件中抽出可测试的章节保存协调器，提供：

- `schedule(snapshot)`：1.2 秒防抖并记住最新快照；
- `flush()`：取消定时器并等待当前请求结束，再保存最新未提交快照；
- `retry()`：保留失败快照和版本，用户重试时继续；
- `dispose()`：清理定时器并触发后台 flush，停止卸载组件的状态回写；同时依靠本地草稿保证页面关闭时仍可恢复。

协调器任何时刻只发一个 PATCH。请求期间继续输入只更新 queued snapshot；请求完成后若 queued 与已保存快照不同，立即发下一次请求，不再等待 1.2 秒。

`update()` 同步把最新草稿写入按用户、小说和章节隔离的 localStorage；保存成功后清除对应本地草稿。读取 `window.localStorage` 属性以及 get/set/remove 都是 best-effort，浏览器禁用存储时不能阻断服务端自动保存。编辑器显示“等待保存、保存中、已保存、保存失败、版本冲突”五类明确状态。失败正文保留在输入框，并提供重试。内部切章和送审先调用 `flush()`；失败时阻止动作。浏览器关闭或刷新使用 `beforeunload/pagehide` 保留并提示尚未落盘的正文。版本冲突只在本地草稿确认清除后重新加载；清理失败时保留当前页和正文并明确提示，避免刷新后反复冲突。

### 3. 编辑状态边界

正文和标题只在 `drafting` 状态可编辑。`review/completed` 状态使用只读输入；用户点击“退回草稿/重新编辑”并成功切换状态后才恢复编辑。送审前必须先 `flush()`，避免检查读取旧正文。

### 4. 正文变化使质量结果失效

Core 更新正文时沿项目既定锁顺序先锁章节、再锁一致性检查项。仅当正文内容实际变化时；只改标题不使正文检查失效：

- 将检查项重置为 `pending` 并清空报告、评分、门禁和返工说明；
- 把该检查项仍为 `pending/running` 的 WorkflowRun 标记为 `cancelled`，错误码为 `QUALITY_SOURCE_CHANGED`；
- 章节保持 `drafting`，不能继续依赖旧终检结果完成。

ReviewArtifact 向 existing_chapter 写入正文时必须复用同一失效逻辑，并把章节退回 `drafting`、清空 `completedAt`。从 `drafting/completed` 重新进入 `review` 时也重置无法证明属于当前正文的历史检查结果；重复提交当前 `review` 状态不破坏同一活动运行。

### 5. 质量运行绑定正文哈希

创建质量 WorkflowRun 时计算当前正文 UTF-8 内容的稳定 SHA-256，并把 `chapterContent` 完整快照、`chapterContentSha256` 与 `sourceUpdatedAt` 写入现有 `WorkflowRun.input`。Agent 上下文读取持久快照，不再临时读取可能已经变化的正文。处理成功或失败回调时都重新计算当前正文哈希：

- 不匹配时将该运行收敛为 `cancelled/failed`，公共检查保持 `pending`；
- 匹配且仍是最新运行时才能写入 completed 报告；
- 旧运行的延迟回调保持幂等，不能覆盖新运行或新正文状态。

跳过操作同样只对当前正文有效；后续正文变化会自动重置 skipped 状态。

### 6. 前端质量状态跟踪

只有 `review` 章节允许创建质量运行；drafting/completed 即使绕过 UI 调用公开接口也返回 409。质量检查受理 202 时公共检查立即进入 running，前端按检查项查询接口轮询 pending/running 状态，使用有上限的退避并在组件卸载或章节切换时取消。终态后刷新完整章节数据。pending/running 时禁用“执行、跳过、标记完成”，轮询失败或超时后展示错误并允许用户继续查询同一运行。活动运行期间拒绝重置/跳过，completed 章节拒绝修改检查状态。

## 错误处理

- 版本冲突保留当前输入并提示重新加载或人工复制，不自动以本地内容覆盖服务端新版本。
- 自动保存失败不改变 `lastSaved`，后续输入继续合并到最新待保存快照。
- 送审 flush 失败时章节仍为 drafting。
- 质量来源变化不是模型失败；公共检查回到 pending，旧运行不能反复重试。

## 测试与验收

- 1.2 秒内切章会先 flush；失败时导航不发生。
- 可控 Promise 逆序场景中始终只有一个请求，最终正文为最新输入。
- 网络失败保留正文和失败状态，重试成功后恢复已保存。
- 服务端 `expectedUpdatedAt` 不匹配返回 409，数据库不变。
- 送审前保存完成，review/completed 正文只读。
- 正文变化会清空 completed/skipped 检查并取消活动运行。
- 版本 A 的延迟质量回调不能放行版本 B；版本 B 的最新运行可以完成。
- 质量 UI 自动显示 pending -> running -> completed/failed，运行中不能重复执行或跳过。
- OpenAPI 客户端重新生成，Web、Core、Ruff、Mypy、类型检查和 Lint 全部通过。
