# 中短篇改纲对话规格

## 状态

- 日期：2026-07-19
- 状态：实现完成
- 范围：中短篇工作区的大纲修改对话、历史回看和版本关联

## 背景

中短篇工作流已经能够保存用户的自然语言修改要求、生成新的完整大纲版本，并允许用户继续批准或修改。但是工作区只提供一次性文本框和独立的版本列表，没有读取 WritingMessage，也没有把修改要求与生成结果关联展示。用户因此看不到自己与 Agent 的改纲过程，功能形态不符合“通过对话反复修改大纲”的要求。

## 目标

- 大纲工作区始终提供独立的“改纲对话”区域。
- 每轮展示用户提交的原始修改要求，不用模型摘要替代用户原话。
- 每轮修改要求关联其产生的大纲 revision，并展示版本号、修改摘要和完成时间。
- 大纲处于 `awaiting_user` 时，用户可以在对话底部继续提交下一轮要求；生成完成后仍停留在同一对话中。
- 已批准、已放弃或已经生成正文的作品仍可只读回看全部改纲对话。
- 旧作品没有 Agent 会话消息时，使用 ReviewArtifact 版本记录补齐结果卡片，不能让历史区域空白。
- 保留完整大纲作为中央阅读对象，对话不替代大纲内容，也不把内部 patch 暴露为主要阅读内容。

## 数据来源与关联规则

- 用户消息读取现有 `GET /api/v1/writing/sessions/{session_id}`，只选择 `intent=revision_focus` 且 `metadata.artifactId` 与当前大纲一致的消息。
- 用户消息中的 `metadata.sourceRevision` 表示提交要求时的大纲版本。正常内容变化后的结果版本为 `sourceRevision + 1`。
- 结果读取现有 ReviewArtifact revision 列表。新版本的 `summary` 作为结果摘要；摘要为空时展示中性文案，不伪造 Agent 回复。
- revision 1 始终作为“根据原始灵感生成初始大纲”的结果事件展示。
- 如果用户要求已持久化但新 revision 尚未产生，根据当前任务状态显示“正在修改”；任务结束且 revision 未增加时显示“已处理，完整大纲内容未变化”。
- 历史数据缺少用户消息时，仍按 revision 展示版本结果；缺少结果消息时不影响用户原话展示。
- 会话消息和版本记录只在前端组装，不新增数据库表、字段或枚举。

## 交互要求

- 对话按时间正序显示，用户消息和版本结果使用不同视觉样式。
- 对话区域有明确标题、当前版本标识和可滚动历史；最新消息滚动到可见区域。
- 提交按钮文案为“发送修改要求”，提交成功后清空输入框。
- 本地存在未保存的直接编辑时，禁止发送自然语言修改，并提示先保存直接编辑。
- 任务执行中禁止重复提交，输入区显示正在处理状态。
- 大纲不是 `awaiting_user` 时隐藏输入控件，显示只读原因，但保留历史。
- 批准与放弃仍是独立决策按钮，不能混在消息列表中伪装成聊天消息。
- 版本历史继续提供详情和恢复；对话中的版本结果可触发同一个版本详情读取。

## 验收

- 连续提交两次改纲要求后，界面按顺序显示两条用户原话及其对应版本结果。
- 刷新页面后对话仍完整存在。
- 已批准的大纲仍能看到历史对话，但不能继续发送修改要求。
- 历史项目只有 revision、没有结果消息时，仍能看到初始版本和后续版本结果。
- 切换到正文或正式正文后再返回大纲，对话能够重新加载。
- 前端相关测试、typecheck 和 lint 通过；PostgreSQL schema 指纹不变。

## 实施结果

- 已新增纯前端对话投影，按 Artifact、`sourceRevision` 和 revision 关联用户原话与完整大纲结果。
- 已新增独立改纲对话组件；`awaiting_user` 时可继续发送，其他状态保留只读历史。
- 已接入 WritingSession 详情读取，刷新页面后重新从 Core 权威消息与版本记录恢复对话。
- 自动化测试覆盖连续两轮、处理中、内容未变化、错误 metadata 和旧项目缺少消息等情况。
- 2026-07-19 在本地开发服务使用真实作品《明天失物招领处》验证：页面展示真实 LLM 产生的“初始版本 → 用户修改原话 → 版本 2 结果”，已批准状态不显示输入框，浏览器无错误日志。
- 验证命令：`npm run test:web`、`npm run typecheck`、`npm run lint` 均通过。
