# 写作操作失败收敛规格

> `sync_lore` 相关设计与验收条目已由 `2026-07-12-remove-sync-lore.md` 取代；当前项目不再提供同步设定流程。

## 目标

让需要草案的快捷操作在模型首次漏交控制事件时获得一次纠正机会，确保等待审核状态能通过 SSE 到达前端，并避免单个复审失败或已上报 Core 的运行异常触发整条写作任务重复执行。

## 非目标

- 不修改 PostgreSQL schema 或现有数据。
- 不增加超过一次的模型纠正重试。
- 不改变草案审核、用户确认和正式写入规则。

## 设计

- 带 `@Agent` 前缀的消息如果同时包含该 Agent 职责内的明确创作动作，应优先采用明确动作；仅在没有匹配动作或动作与指定 Agent 职责不一致时，才回退到该 Agent 的默认操作。保留的前端快捷入口必须分别稳定路由为 `plan_chapter` 和 `rewrite_scene`。
- 主责智能体首次未提交合法草案控制事件时，追加明确的控制工具要求并仅重试一次；第二次仍缺失时维持现有错误结果。
- 同一响应同时包含通用草案事件和当前 Operation 的专用草案事件时，优先采用专用事件；`plan_chapter` 必须优先使用 `submit_beat_plan`，不能被较早出现的 `begin_artifact_output` 错误接管。
- 纠正重试解析草案时合并前后两次运行的可见内容和控制事件，使更新构建器可以从第一次的开始事件继续到第二次的完成事件。
- 更新构建器在单次 AgentRuntime 中只能启动一次；启动后不再向模型暴露 `start_update_builder`。重复启动同一 `artifactKey` 不得产生新的控制事件，追加和完成操作必须使用已启动的同一 `artifactKey`。
- 当前运行创建 ReviewArtifact 后，Agent Service 将已提交 Core 的完整草案请求作为权威复审上下文注入 reviewer，并只暴露控制工具。复审不得再依赖当前尚未注册的读取工具；跨进程恢复时如果本地没有该上下文，仍保持现有显式失败收敛，不猜测草案内容。
- `propose_updates` 与 `finish_update_builder` 成功后立即结束设定 Agent 的本轮工具循环。更新构建器的工具结果必须明确告知模型当前阶段和下一步，不得只返回无状态的统一确认。
- 跨纠正重试合并构建器事件时，相同 `artifactKey` 的重复开始事件按幂等处理，不得清空此前已经追加的批次。纠正提示必须明确要求一次完整的 `start → append → finish` 生命周期。
- 父图中的 Operation 子图触发用户确认中断时，以 `__interrupt__` 中的草案标识补全外部稳定状态，确保任务处理器能生成 `waiting_user` 扁平快照，并支持后续由新的 Agent 实例恢复。
- 图进入等待用户确认时，Agent Service 先发送 `artifact_awaiting_user_approval`，再保存带有最新事件序号的稳定 checkpoint。
- 浏览器消费标准 SSE 时，以 `event:` 行作为事件类型，并兼容旧载荷中的 `data.type`；不能只读取 `data.type` 而丢弃 Agent Service 事件。
- `artifact_awaiting_user_approval` 到达浏览器后应结束当前 HTTP 事件流，使前端释放发送锁；WritingTask 本身继续保持可恢复的等待确认状态，不发送伪完成事件。
- 复审智能体执行异常时生成 `block` 复审结论，使草案继续进入用户审核，不让单个复审故障中断整条运行。
- 写作处理器已经向 Core 上报失败后，向队列返回不可重试错误；队列直接确认失败，不再重复执行可能已经产生草案等副作用的图。
- 图稳定结束于 `phase=error` 时，写作处理器必须保存错误快照并调用 Core 失败回调，禁止调用完成回调。

## 验收标准

- “规划本章”“重写场景”的固定提示即使带有 `@剧情`、`@写作` 前缀，也必须进入对应的明确 CreativeOperation，不能退回 Agent 默认操作。
- 父图中断结果即使没有合并子图状态，也必须从中断载荷恢复草案标识和 `waiting_user` 阶段。
- 章节规划同时出现 `begin_artifact_output` 与 `submit_beat_plan` 时，必须使用结构化章节计划事件生成草案。
- `event: artifact_awaiting_user_approval` 且 `data` 仅包含草案载荷时，前端必须识别为待确认事件并刷新草案。
- 待确认事件发出后，本次 SSE 流必须正常结束，前端快捷操作不能持续处于禁用状态。
- 章节规划和设定同步首次漏交控制事件、第二次正确提交时可生成草案。
- 更新构建器跨纠正重试的事件可以组成完整草案。
- 设定 Agent 启动更新构建器后，后续模型轮次看不到 `start_update_builder`，并能从工具结果得知应继续追加或完成。
- 同一运行内的复审 Agent 直接收到完整草案上下文，只调用评审控制工具，不会因读取草案及作品工具未注册而循环。
- 相同 `artifactKey` 的重复开始事件不会覆盖此前追加的更新；完成事件仍可生成完整草案。
- `propose_updates` 或 `finish_update_builder` 成功后，设定 Agent 不再进入额外模型轮次。
- 等待用户确认时，前端可通过 SSE 收到草案标识并刷新待确认入口。
- 单个复审异常不会令整条图抛出异常。
- 已上报 Core 的写作失败只执行一次。
- `phase=error` 的写作任务只调用失败回调，不调用完成回调。
- 不修改数据库结构。
