# Agent 辅助任务工作流日志

## 背景

文风画像和章节质量检查会调用 `ModelRuntime`，其观察器要求当前 `runId` 已存在人工工作流日志。写作任务会在进入模型流程前创建日志，但画像和质量检查此前没有建立同一生命周期，导致模型已经返回后，日志观察器因找不到运行记录而抛出异常，任务最终被错误标记为失败。

## 目标

- 文风画像和章节质量检查在首次模型调用前创建人工工作流日志。
- 日志使用队列任务中的 `runId`、`taskId`、`userId` 和 `novelId`，不得读取数据库或绕过 Core API。
- 任务成功时记录“完成”，模型、工具或 Core 回调失败时记录“错误”。
- `WorkflowModelObserver` 保持严格：日志写入本身失败时继续显式失败，不静默吞掉异常。
- 不修改 PostgreSQL schema、服务契约、队列负载或计费规则。

## 实现约束

- 画像和质量检查处理器接收与写作处理器相同语义的可选工作流日志端口，单元测试可以不注入。
- 画像日志运行类型为“文风画像”，质量检查日志运行类型为“质量检查”，二者的 `chapterId` 暂记为 `null`，因为当前队列契约不携带章节标识。
- 日志必须先于模型运行时观察器写入；终态必须在成功回调后或失败回调前后可靠记录。

## 验收

- 使用真实 `FakeModelProvider`、`ModelRuntime`、`WorkflowModelObserver` 和 `HumanWorkflowLog` 的画像处理器测试能够完成五个画像分节，不再抛出“运行日志不存在”。
- 质量检查处理器测试验证开始与成功/失败终态。
- Agent Service 相关 pytest、Ruff 和 Mypy 通过。
- 三服务联调中，文风画像和章节质量检查主流程通过。
