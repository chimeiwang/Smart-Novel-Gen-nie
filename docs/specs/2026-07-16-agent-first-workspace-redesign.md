# Agent 优先工作区重设计规范

## 背景

当前工作区把章节、编辑器、资料面板和智能写作入口并列呈现，AI 创作任务、待确认产物、阅读与小修之间缺少稳定的主次关系。会话默认选择、章节链接、计划空态和质量终检完成态也存在展示与服务端门禁不一致的风险。本次重设计把工作区收敛为以 Agent 任务为主线、作者确认正式产物的创作控制台。

## 当前项目事实

- 工作区由 Next.js 页面、章节列表、章节编辑器、资料延迟加载器和写作会话组件组成。
- `lore`、`planning`、`resources` 三组延迟接口和前端缓存已经存在，可以按需复用。
- 写作会话列表与会话详情、自动路由 CreativeOperation、ReviewArtifact 审核链路均已有公共接口。
- 章节编辑器使用 `textarea` 和 1.2 秒自动保存；章节导航前已有 flush 保存入口。
- 公共质量检查 DTO 已包含 `status`、`result`、`scoreOverall` 和 `qualityGate`。一致性报告的合法门限只允许 `pass` 或 `revise`。
- Core 当前章节完成门禁只判断质量检查状态是否为 `completed` 或 `skipped`，尚未验证 completed 结果是否完整。

## 目标

- 默认进入 Agent 优先的创作工作室，并提供稳定的阅读与资料视图。
- 让每章恢复最需要继续处理的会话，历史会话仍可查找且标题可读。
- 让所有待确认 Agent 产物始终出现在固定审核栏，不因会话或视图切换丢失。
- 统一前端展示与 Core 完成门禁对“有效终检”的判断。
- 保留现有公共 API、数据库结构和延迟加载边界。

## 非目标

- 不新增、删除或修改公共 API、生成客户端字段或 PostgreSQL schema。
- 不改变 Agent Service、CreativeOperation 自动路由、ReviewArtifact 状态机或正式数据应用流程。
- 不把阅读模式变成新的富文本编辑器，不替换章节 `textarea`。
- 不合并三组延迟接口，不预加载完整 workspace 聚合数据。

## 设计方案

### 视图与 URL

工作区使用 `view` 查询参数，唯一合法值为 `studio`、`reading`、`library`。参数缺失、数组值或非法值都回退到 `studio`。章节链接必须同时保留目标 `chapterId` 与当前合法 `view`，避免章节切换后意外回到默认视图。

- `studio`：默认视图，以 AI 创作任务、Agent 运行状态和产物审核为主，正文编辑为当前任务提供上下文。
- `reading`：默认只读，突出连续阅读；只有用户显式点击“小修”后才进入编辑状态。
- `library`：集中浏览创作资料，沿用 `lore`、`planning`、`resources` 三组及其现有延迟接口、缓存、失效和重试行为。

`library` 不复用章节侧栏的临时标签页，而使用独立的创作资料工作区。左侧资料导航严格按以下三组组织，中央区域显示当前分类的完整内容或表单：

- 设定：角色、地点、势力、物品、术语、故事背景、世界设定。
- 故事规划：大纲、剧情进度、故事进展。
- 写作规则与素材：作品圣经、文风、参考资料。

资料模式未进入时不得请求任何资料接口；进入资料模式后也不预加载全部数据，只有用户进入具体分类时才加载该分类需要的组。角色、地点、势力、物品、术语只加载 `lore`，故事背景、世界设定、大纲、剧情进度、故事进展和作品圣经只加载 `planning`，文风和参考资料只加载 `resources`。加载结果按现有加载器缓存；同组失效后，当前分类重新加载该组，失败时允许只重试该组。如果未来一个界面分类同时需要 `planning` 和 `resources`，可以在进入该分类时同时按需加载两组，但不得因此预加载其他分类的数据。

大型表单使用资料主画布内的全宽分区详情，不在狭窄侧栏弹窗中堆叠多个完整表单。设定详情可继续复用 `LorePanel`，但其编辑覆盖层必须相对资料主画布定位，不得写死旧三栏的 `left`、`right` 或顶部尺寸。

三种视图共享同一个已挂载的章节会话状态。切换视图只改变可见布局，不能卸载或重建当前写作会话、丢失流式状态、草稿输入或审核上下文。用户离开 reading 小修、切换章节或执行会导致编辑区离开的导航前，必须先调用现有 flush 保存；保存失败时阻止离开并显示错误。

本次验收的导航保护范围仅包括应用内模式按钮和章节链接。浏览器原生前进、后退可同时改变章节历史位置，不能由客户端异步 flush 可靠阻断；该路径依赖现有自动保存与页面生命周期保存，不做伪回滚或只恢复部分 URL 参数。Next 路由完成历史切换后，客户端外壳以新的 `initialView` 同步可见视图。

### 桌面宽度与阅读尺度

完整桌面工作区的最小设计宽度为 1440px，日常重点优化宽度为 1920px，最终以 `1440x900` 和 `1920x1080` 两组视口验收。1440px 下必须完整保留章节/资料导航、中央主画布和固定审核栏三栏，参考列宽约为 `240-280px / minmax(0, 1fr) / 340-400px`；不能在 1280px 以上提前把审核栏折到下一行。1920px 下新增空间由中央主画布吸收，侧栏不随视口无限增宽；章节正文和 Agent 对话保持可读最大宽度，避免单行过长。小于 1440px 可以通过水平滚动或降栏等方式降级，但不能反向限制 1440px 和 1920px 的完整布局。

### Agent 主线与审核栏

`studio` 的首要动作是描述创作任务。前端提交自然语言任务，由现有 CreativeOperation 机制自动路由主责 Agent；Agent 选择器只作为已有能力配置，不取代自动路由。Agent 产物继续遵循 `proposal -> ReviewArtifact -> 复审/返工 -> 用户确认 -> Core API 应用`。

所有 `awaiting_user` 产物显示在工作区固定审核栏。审核栏不依赖聊天滚动位置，不因 `studio`、`reading`、`library` 切换而消失；点击后仍使用现有审核详情和决定接口。正式内容只能在用户确认后由 Core 应用。

### 章节会话恢复与标题

每章打开时只依据现有会话列表 DTO 选择默认会话，不为选择过程逐条请求会话详情。先筛选 `phase` 为 `discussing`、`generating` 或 `recording` 的活动会话，候选超过一个时选择 `updatedAt` 最新者；没有活动候选时选择该章 `updatedAt` 最新的会话；没有会话时保持空态，首次提交任务再按现有接口创建会话。等待用户处理 ReviewArtifact 的会话由列表中的 `recording` 阶段表示。默认会话选定后才请求其详情，并从详情恢复 `currentTask`、消息和待审核上下文。其他会话进入历史列表，不删除、不静默改绑历史任务。

会话显示标题只使用列表 DTO，按以下顺序生成：非空 `title`；非空 `lastMessage.content` 摘要；“未命名会话”。摘要先合并连续空白，再按展示上限截断并追加省略号。新会话由创建动作传入当前任务标题，不要求列表接口提供首条消息。该规则只影响历史列表展示，不写回数据库。

### 章节计划与 pending 状态

章节列表只为当前章节显示 `approvedBeatPlan=null` 的“未确认章节计划”提示。非当前章节没有已批准计划时返回空展示，不以“未确认”暗示存在待处理产物；存在已批准计划时仍显示场景数和预计字数。

新增章节与章节导航使用相互独立的 pending 状态。创建中只禁用创建入口，导航中只标记并禁用对应导航，避免一个动作让整份章节列表失去响应。两条路径在离开当前小修内容前都必须先 flush。

### 质量终检展示与完成门禁

有效的 `completed` 一致性终检必须同时满足：

1. `status === "completed"`；
2. `result` 去除首尾空白后非空；
3. `scoreOverall` 是有限数值；
4. `qualityGate` 为 `pass` 或 `revise`。

`skipped` 是独立的“已处理”状态，不伪造成有效 completed 报告。字段不完整的 `completed` 必须显示为 `invalid`，不得显示“已完成”、不得提供报告成功态，也不得满足章节完成条件。`rewrite` 不是当前一致性报告契约的合法门限。

Core 章节完成门禁必须复用同一判断：`skipped` 可以完成；`completed` 只有在上述报告字段完整时才可以完成；无效 completed 返回明确冲突错误。前端只做一致展示和提前禁用，Core 仍是最终权威。

## 影响范围

- `apps/web`：工作区查询参数和链接、稳定视图容器、会话默认选择与标题、章节计划展示、创建/导航 pending、质量结果展示。
- `apps/core-api`：章节完成门禁校验，但不改变请求或响应契约。
- 既有 Agent Service、service contracts、OpenAPI 客户端、数据库 schema 和延迟接口保持不变。

## 验收标准

- `parseWorkspaceView` 只接受三个合法值，其他输入回退 `studio`；章节链接保留 `chapterId` 和当前视图。
- 通过应用内模式按钮或章节链接切换时，当前会话、流式任务、输入和待确认产物不重置；离开前完成 flush，失败时不导航。浏览器原生历史切换不伪装为可阻断导航，完成后以 Next 下发的 `initialView` 同步外壳。
- 每章仅依据列表 DTO 优先选择 `discussing`、`generating`、`recording` 活动会话，否则选择最近会话；选中后再加载详情恢复任务。历史会话标题只按显式标题、最后消息摘要和未命名空态降级。
- `library` 继续按 `lore`、`planning`、`resources` 三组延迟加载。
- `library` 使用独立 `LibraryPane`，三组与十三个具体分类名称、顺序和归属严格符合规范；未进入资料模式不请求，进入具体分类只加载该分类需要的数据，缓存、失效和同组重试继续有效。
- 大型资料表单在中央主画布全宽详情中呈现；设定编辑层不包含旧布局的固定 `left`、`right` 坐标。
- `1440x900` 下完整显示三栏，列宽保持约 `240-280px / minmax(0, 1fr) / 340-400px`；`1920x1080` 下中央区域吸收多余空间，正文和对话仍受阅读宽度约束；小于 1440px 有明确降级规则。
- 非当前章节的空计划不显示“未确认章节计划”，当前章节仍显示该提示。
- 新增章节与章节导航 pending 互不阻塞。
- 只有完整的 completed 报告或 skipped 状态满足前端已处理展示与 Core 章节完成门禁；无效 completed 显示 `invalid`。
- 前端纯逻辑测试先以缺少实现模块或行为未实现的原因进入 RED，后续实现不得通过弱化断言绕过这些契约。
