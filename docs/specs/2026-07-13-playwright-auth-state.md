# Playwright 认证状态复用与文风流程验收规格

## 背景

本地端到端测试通过同一个 Next.js 开发代理访问 Core API。Core 对注册接口执行每来源每小时三次的安全限流，而现有六个 Playwright 主流程分别注册随机用户，因此整组测试必然在第四次注册时收到 `429 RATE_LIMITED`。此外，文风库在创建文风后会自动展开新条目，旧测试仍等待并点击“展开”，与当前 UI 行为不一致。

## 目标

- 一次 `npm run test:e2e` 只注册一个随机测试用户。
- 注册、退出和重新登录流程继续作为独立验收场景执行。
- 登录成功后的浏览器认证状态保存到 `output/playwright`，其余五个业务场景通过 Playwright 项目依赖复用该状态。
- 文风场景按“创建后自动展开”的当前产品行为验证上传与画像生成。
- 六个主流程仍由单个 Chromium worker 串行运行。

## 非目标

- 不提高、关闭或绕过 Core 注册限流。
- 不信任测试传入的转发来源头，也不新增测试专用认证接口。
- 不删除或重置共享 Redis 中的认证限流键。
- 不修改 PostgreSQL schema、迁移或正式认证业务规则。
- 不通过延长固定等待时间、减少断言或跳过业务步骤让测试通过。

## 设计

### 认证准备项目

`auth.spec.ts` 保留“注册、退出并重新登录”的完整流程。最终重新登录成功后，测试把当前浏览器上下文写入 `output/playwright/.auth/user.json`。

Playwright 配置新增认证准备项目，并让 Chromium 业务项目依赖该项目。认证准备项目只运行 `auth.spec.ts`；业务项目忽略该文件并加载上述 `storageState`，因此其余五个场景进入页面时已经登录，不再调用注册接口。

认证状态属于测试产物，不得提交 Git。每次完整测试仍创建一个新的随机用户，以继续验证真实注册流程，并避免依赖预置账号或固定密码状态。

### 业务场景

项目编辑、知识与文风、质量检查、写作草案应用和草案丢弃场景移除各自的注册步骤，但继续创建独立小说和业务数据，避免场景间共享小说状态。

文风场景创建文风后必须先确认新条目处于展开状态，再向可见的文件输入控件上传资料。测试不得再次点击“展开”，因为组件在创建成功时已经自动展开新条目。

项目编辑场景打开创建弹窗后，提交按钮必须限定在创建表单内，避免与页面头部同名的“新建小说”入口产生严格模式歧义。

写作草案应用和丢弃场景使用当前写作面板的“生成正文”快捷动作。该动作会在没有当前会话时自动创建会话并启动正文草案流程；测试不再依赖已经移除的“智能写作”入口，也不重复手工创建会话。

文风名称必须带本次运行的唯一后缀，避免持久测试数据中的同名文风使严格定位产生歧义。质量检查由 UI 启动后，通过公开 workspace API 轮询异步终态；任务完成后刷新当前页面、重新打开终检面板并验证“已完成”，不使用固定睡眠，也不假设页面具备 SSE 自动刷新能力。

写作上下文要求当前章节唯一映射到结构化章节组。两个写作草案场景必须通过公共大纲节点接口建立覆盖第 1 章的最小 `stage → plot_unit → chapter_group` 层级，再启动正文生成；不得绕过或放宽 Core 的 `CHAPTER_GROUP_MAPPING_CONFLICT` 校验。

质量检查的零扣费断言必须读取 `/api/v1/billing/usage` 的 `totalUsage.totalTokens`；账单摘要接口只负责余额和流水，不提供 token 汇总字段。写作草案应用与丢弃不得依赖操作后已经移除的瞬时提示文本：应用场景轮询工作区正文，丢弃场景验证待确认数量归零且章节正文未被修改。

## 错误处理

- 认证准备失败时，依赖项目不得继续执行，避免产生一组无意义的未认证失败。
- `storageState` 写入失败必须直接使认证场景失败，不得退回到每个业务场景重新注册。
- 业务 API 或画像生成失败继续由现有断言和 Playwright trace、截图、视频报告，不吞掉错误。

## 验收

- `npm run test:e2e:list` 仍列出六个主流程，不重复执行认证用例。
- 一次完整运行最多调用一次注册接口。
- 注册、退出、重新登录用例通过。
- 其余五个场景使用已保存认证状态执行，并且不调用 `registerWithApi()`。
- 文风场景不再等待不存在的“展开”按钮，仍覆盖文件上传和画像生成。
- 项目编辑场景通过创建表单内的提交按钮创建小说。
- 两个写作草案场景通过“生成正文”快捷动作进入待审核草案状态。
- 文风场景使用唯一名称，质量检查场景轮询公开业务状态后回到 UI 验证结果。
- 写作场景先创建最小合法结构化大纲，并验证草案应用或丢弃。
- Core 注册限流实现与策略保持不变。
- 相关 Web 测试、类型检查和 Lint 通过。
