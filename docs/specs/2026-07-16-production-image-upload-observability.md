# 生产镜像上传可观测性与超时规范

## 状态

- 日期：2026-07-16
- 状态：已实现
- 范围：GitHub Actions 生产镜像上传、远端容量预检、阶段日志和超时边界

## 背景

当前上传脚本把多张镜像的 `docker save`、压缩、SSH 传输和远端 `docker load` 合并为一条流水线。流水线运行期间没有阶段进度、镜像大小或明确超时；当网络、远端磁盘或远端 Docker 变慢时，只能看到上传步骤长时间运行，无法判断卡在哪张镜像、哪个阶段。

2026-07-16 的生产部署中，镜像复用检查已经明显变慢，随后两张镜像的合并上传在约 55 分钟内没有产生完成日志，最终只能人工取消。应用测试和镜像构建均已完成，远端部署脚本尚未启动，因此需要先改造上传边界，而不是调整应用实现。

## 目标

- 在传输前检查远端 Docker 可响应、Docker 数据目录可用空间和临时目录可用空间。
- 每张不可复用的镜像单独归档、压缩、传输和导入。
- 日志明确显示当前镜像、当前阶段、压缩后大小和阶段耗时。
- 为单张镜像传输导入和整个上传步骤设置有界超时。
- 任一镜像失败时立即停止，不启动生产版本切换，并在日志中指出失败镜像和阶段。
- 临时归档只保存在一次性 GitHub Runner，并在成功、失败或取消时清理。

## 非目标

- 不引入第三方镜像仓库。
- 不在生产服务器构建镜像。
- 不修改 PostgreSQL schema，不执行数据库迁移。
- 不自动清理生产服务器镜像、容器、卷或数据。
- 不降低 SSH 主机身份校验要求。

## 设计

### 远端预检

首次读取或上传镜像前，通过现有严格主机校验的 SSH 连接执行只读预检：

- SSH 必须使用非交互模式，并在远端 shell 启动后立即输出阶段标记；
- `docker info` 必须在限定时间内成功；
- 远端必须提供支持 `pipefail` 的 Bash；
- 输出 Docker 根目录所在文件系统的可用容量；
- 输出远端临时目录所在文件系统的可用容量；
- 不输出环境变量、密钥或生产配置内容。

预检只能提供诊断和提前失败，不根据容量结果自动删除任何服务器资源。

### 逐镜像上传

对于每张无法安全复用的镜像，按固定顺序执行：

1. 在 Runner 临时目录生成该镜像的压缩归档；
2. 输出镜像名、压缩后字节数和压缩耗时；
3. 按本地镜像逻辑大小的 2 倍再加 512 MiB 安全余量，检查远端 Docker 数据目录的可用容量，以覆盖导入时解包层和最终层并存的峰值；
4. 通过标准输入把单个归档传给远端 `docker load`；
5. 远端解压和导入使用 `pipefail`，任一环节失败都必须返回非零；
6. 对归档阶段、传输和远端导入的组合阶段分别设置单镜像超时；
7. 成功后输出导入耗时并删除本地临时归档；
8. 失败时输出镜像名和失败阶段，由退出陷阱清理剩余临时文件。

使用较低压缩级别，避免 GitHub Runner 为节省少量网络流量付出不成比例的压缩时间。归档不落到生产服务器临时文件，避免额外占用 2 核 2 GB 主机的磁盘峰值。

### 超时边界

- SSH 建连设置明确的连接超时和有限重试次数；
- 远端只读查询、Docker 预检和镜像存在性检查设置命令级超时；
- 所有超时参数必须是限定范围内的正整数，超时后先终止进程组，再用强制终止兜底；
- 服务器明确返回“镜像不存在”时才允许进入上传路径，SSH、Docker 或查询超时必须立即失败；
- 每张镜像的传输与导入设置独立超时，超时错误必须包含镜像名；
- GitHub Actions 的“上传 Docker 镜像”步骤设置总超时，作为脚本级超时之外的最后边界。

超时只终止当前发布，不触发版本切换或自动清理生产资源。

## 错误处理

- 本地归档失败：报告具体镜像和归档阶段，停止上传。
- SSH 或远端 Docker 预检失败：报告预检阶段，停止上传。
- 单镜像传输或导入超时：报告具体镜像、超时秒数和传输导入阶段，停止上传。
- 远端 `docker load` 返回非零：保留其标准错误，报告具体镜像，停止上传。
- 工作流总超时：GitHub Actions 明确把上传步骤标记为超时失败，不继续执行远端部署。

## 测试与验收

- 架构测试证明上传步骤存在总超时。
- 脚本测试证明不再把多张镜像交给一次 `docker save`。
- 脚本测试证明每张待上传镜像单独归档、单独调用远端导入并输出阶段日志。
- 脚本测试证明远端预检在上传前执行。
- 脚本测试证明单镜像传输导入失败会返回非零，且错误包含镜像名和阶段。
- Shell 语法检查和现有部署安全测试通过。
- Pull Request 的 CI 成功后合并到 `main`；`main` 的生产工作流中，上传日志能显示逐镜像阶段，且整个发布成功或在有界时间内给出可定位失败。
